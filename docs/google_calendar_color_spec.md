# Googleカレンダー 配色仕様・実装設計ガイドライン

GoogleカレンダーのUI構成、配色ルール、およびシステム連携時におけるフロントエンド実装の注意点をまとめた仕様ドキュメントです。

---

## 1. 画面構成パーツと配色ルール

GoogleカレンダーのUI（特にWeb版ライトモード）は、以下の4つのレイヤーで背景色やアクセントカラーが管理されています。

| パーツ・エリア                 | 標準のカラーコード（目安）       | 役割とシステム上の特徴                                                                                |
| :----------------------------- | :------------------------------- | :---------------------------------------------------------------------------------------------------- |
| **全体の背景（キャンバス）**   | `#FFFFFF` （純白）               | スケジュールを表示するメインのグリッド背景。                                                          |
| **ヘッダー・サイドバー**       | `#F8F9FA` 〜 `#F1F3F4`           | メイン背景と視覚的に区別するための薄いグレー。ボタンやミニカレンダーが配置されるエリア。              |
| **今日（Today）の強調表示**    | `#1A73E8` （Google Blue）        | 当日の日付（円形塗りつぶし）や、タイムライン上を走る「現在時刻線（Current Time Indicator）」に使用。  |
| **イベント枠（予定ボックス）** | _選択したイベントカラーに基づく_ | **【重要】** 予定のタイプ（終日か時間指定か）によって、背景色と文字色の適用ロジックが動的に変化する。 |

---

## 2. イベントカラー（予定ボックス）の動的配色ロジック

外部システムでGoogleカレンダーの見た目を再現、あるいは同期表示する際、最も手戻りが発生しやすいのがこの**「文字の視認性（コントラスト比）の自動計算」**です。Googleは以下の2つのロジックを使い分けています。

### ① 終日イベント（背景が濃いベタ塗り）

- **背景色**: ユーザーが選択したカラー（例：Tomato `#DC2127`）がそのまま100%の濃度で適用。
- **文字色**: 自動的に **`#FFFFFF`（白）** に固定。

### ② 時間指定イベント（背景が薄いパステル調）

- **背景色**: 選択したカラーの明度を上げ、彩度を下げた（あるいは透過度を上げた）**「パステル調の薄い色」**が適用。
- **文字色・左端の線**: 元の濃いカラー（例：Tomato `#DC2127`）がそのまま適用。

> ⚠️ **致命的なバグ・技術的負債アラート**
> 外部システムの実装で、単に「APIから取得したカラーコードをそのままイベントの背景色（`background-color`）にし、文字色を黒（`#000000`）に固定する」という処理を行うと、グラファイトやブルーベリーなどの濃い色において**文字が背景に埋もれて完全に読めなくなる（アクセシビリティ違反）**が発生します。

## 3. 実装アプローチにおける3つの戦略的選択肢

カレンダー連携機能やUIを開発するにあたり、リリース速度、運用コスト、再現度のトレードオフから選択すべき3つのアプローチを提示します。

### 選択肢A：Google公式（Material Design 3）の完全模倣

- **概要**: 本家GoogleカレンダーのCSS設計やカラーパレット、ダークモード時のリマップ挙動を完全に再現する。
- **メリット**: ユーザーにとって本家と全く同じ違和感のないUXを提供できる。
- **デメリット**: Google側のデザインアップデートに伴う追従コスト（運用の保守コスト）が極めて高い。

### 選択肢B：カラーIDマッピングによるCSS変数管理（★推奨）

- **概要**: Google Calendar APIから返される `colorId`（1〜24）をトリガーにし、自社システム側で定義した24色のライト/ダーク用カラーパレットをCSS変数（例: `var(--calendar-color-1)`) で制御する。
- **メリット**: 最も開発スピードが速く、低コスト。ダークモードや多言語対応時の手戻りリスクを最小限に抑えられる。
- **デメリット**: 本家Googleカレンダーの動的な透過率やグラデーションと100%一致させるのは難しい。

### 選択肢C：カスタムカラー（Hex）の完全同期

- **概要**: ユーザーがGoogle側で設定した自由な16進数カラーコード（Hex）をそのままシステム側でも動的に受け付ける。
- **メリット**: 自由度が最も高く、ユーザー個別のカラーカスタマイズを完全に再現可能。
- **デメリット**: 渡された背景色に対して「文字色を白にするか黒にするか」を判定するコントラスト自動計算ロジック（WCAG基準に準拠した輝度計算式）を自社側で実装・テストする必要があり、工数が肥大化する。

## 4. 開発時のチェックリスト（軍師のアラート）

1. **APIの仕様差異の把握**
   - Google Calendar APIでイベント色を指定・取得する際は、原則カラーコードではなく、`colorId`（文字列の数値）を使用する設計にすること。
2. **ダークモードへの考慮**
   - Googleカレンダーは端末のモードに応じて動的にコントラストを調整（動的リマップ）しているため、カラーコードのハードコーディングは原則避ける。
3. **アクセシビリティ（視認性）の担保**
   - 時間指定イベントの背景を薄くするロジック、または文字色を反転させるロジックが仕様に含まれているかを必ず確認すること。
