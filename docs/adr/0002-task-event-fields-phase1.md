# ADR-0002: 詳細入力項目の拡充（フェーズ1）と「整理を生まない」原則

- ステータス: 採択（Accepted・方針A で改訂）
- 日付: 2026-06-18
- 関連: `task-board-spec-v1.md`（§3.3 タスク詳細）、`supabase/migrations/20260618000000_task_event_fields.sql`

## 背景（Context）

side-peek 詳細モーダルの再検討で、一般的なタスク管理／予定入力に対する項目不足を洗い出した。
当初フェーズ1として「①タスク優先度（高/中/低）②締切時刻③予定の色分け」を実装したが、レビューで
**「色や優先度の設定そのものが"整理作業"になり、Ichimoku の北極星（整理せず消化する）と矛盾する」**
という指摘を受けた。

## 決定（Decision）

**方針A（思想優先・整理を生まない）を採択。**

1. **採用: タスクの締切時刻**。`tasks.due_time`（`time`・任意）。`due_date` がある時のみ意味を持つ。
   消化の判断（今日の何時までか）に直結し、分類のための作業を生まない。
   - 締切設定時、日付入力の隣に時刻入力（空＝日付のみ）。締切カラム・カード・カレンダーの締切表示に
     `M/D HH:mm` で併記。
2. **不採用: タスクの優先度**。★（対応中）と締切緊急度で焦点は足りており、優先度付けは整理作業を増やす。
3. **不採用: 予定の手動色分け**。ユーザーが色を選ぶ行為自体が整理作業。手動の色は持たせない。

## 「整理を生まない」原則（この ADR で確立）

詳細項目を増やすときは次を満たすこと:

- **既定で無設定でも成立する**（必須入力にしない）。
- **分類・整理のための作業を強要しない**。色・ラベル・優先度のような"仕分け"系は原則持たせない。
- **消化（次に何をやるか／いつまでか）に直結するシグナルだけ**を、できれば自動で与える。

予定とタスクの視覚的区別が必要な場合も、**ユーザーに色を選ばせるのではなく、種別から自動で**
描き分ける（手作業の整理を発生させない）。

## 影響（Consequences）

- マイグレーション `20260618000000_task_event_fields.sql` は `tasks.due_time` のみ追加に縮小。
  実 DB 未適用（このリモート環境に Supabase 認証情報なし）。既存行は既定値で後方互換。
- 「繰り返し」はフェーズ1に含めない。DB の `rrule`/`recurring_event_id`/`is_exception` 箱は既存のまま、
  実装は Google Calendar 連携（Step 12-13）と同時に行う。
- サブタスク／チェックリスト（フェーズ2）も「整理を生まない」原則の観点で要再評価。
