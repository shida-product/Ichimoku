## 0. 北極星（設計の最上位原則）

**考えさせない・忘れさせない・整理させない。**
注意が逸れやすい状況でも回るよう、記憶や意志力に頼らせない。システム側が「今やること」と「迫る締切」を勝手に前へ押し出す。
→ この原則に反する機能（重い整理・手入力の多い計画機能）は採用しない。

---

## 1. コンセプト

1画面に3つの面を並置し、画面遷移なしで完結する：

1. **タスクボード**（左・メイン）… カテゴリ × 状態の2次元ボード。作業を個別登録 → ドラッグで消化。
2. **近日締切レーン**（上・常時表示）… 締切のあるタスクを締切順に常に見せ、埋もれを防ぐ。
3. **カレンダー**（右）… 接待・講演などの固定予定。v1はアプリ内で作成・編集まで。

タスク（こなす作業）と予定（出席する固定枠）は**別物として扱う**。締切は時間軸（カレンダー）に埋めず、常時表示のレーンで管理する。

### 1.1 差別化の中核 ★v1.2

**「すべてを1画面に、画面遷移なしで」** がこのプロダクトの存在意義。
既存ツール（Google Tasks / Todoist 等）は "リスト⇄ボード⇄カレンダー" を**タブで切り替える**設計。本プロダクトは切り替えを廃し、ベース画面を固定したまま情報を**スライドイン／ポップで重ねる**（§3.7）。
→ 新規性ではなく**「遷移ゼロで全状況が一目」というフィット**で価値を出す。注意が逸れやすいほど、画面遷移＝忘れる隙になるため、ここを最重要要件とする。

## 3. 機能要件（v1）

### 3.1 認証

- メール＋パスワード or マジックリンク（Supabase Auth）
- 自分のデータのみ閲覧・編集可（RLSによる完全なマルチユーザー隔離）
- **マルチユーザー設計**:
  - 全ユーザーで同一のデータベースおよびテーブルを共有する。
  - 新規アカウント登録時に、Supabaseが `auth.users` テーブルに自動で一意のUUID（`auth.uid()`）を作成する。
  - すべてのテーブルのデータは `owner_id` フィールドにそのUUIDを保持し、RLSポリシーによって他人のデータには絶対にアクセスできないように制御される。これにより、ユーザーごとにデータベースを分ける必要なく、完全に独立した個人専用画面として機能する。

### 3.2 カテゴリ

- 作成 / リネーム / 並べ替え / 削除（ユーザー定義：事務・経営・採用…）
- ボード上ではスイムレーン（横の帯）として並ぶ
- **カテゴリは折りたためる**（視界を絞れる）

### 3.3 タスクボード（カテゴリ単位の単一リスト）★v1.4 改訂

> **v1.4 改訂（画面構成見直し / ADR-0001）**: 「整理しないで消化する」ゴールに対し、`未着手`/`対応中`の状態列分割は"整理作業"を強いるため廃止。状態列をやめ、**カテゴリごとの単一リスト**にした（案B）。

- 縦＝カテゴリ（スイムレーン）。各レーンは**1本のタスクリスト**（状態で列分割しない）。
- **「対応中」はカード上の★フラグ**で表す（列ではない）。★クリックでトグル（内部状態 `doing`）。
- タスクの作成（クイック追加）/ 編集 / 削除
- 属性：タイトル（必須）、メモ（任意）、**リンク（任意・複数可）**、カテゴリ（任意）、対応中フラグ、**締切（任意・日付＋任意の時刻）★v1.5**、並び順
- DnD：
  - **レーンを跨ぐ移動** = カテゴリ変更
  - **レーン内の上下移動** = 並べ替え（`position` を fractional index で永続化）
  - **下部の共有「完了ゾーン」へドロップ** = 完了（消化）
- **完了＝即アーカイブ**：完了ゾーンにドロップした瞬間に `completed_at` と `archived_at` を記録し、ボードから即座に消える。誤操作救済として数秒の取り消し（undo）トーストを出す。
- **完了履歴**：完了したタスクは別 side-peek「完了履歴」に**新しい順**で残り、そこから参照・未着手へ戻せる。**完了から30日経過で DB からも物理削除**（N=30 に確定 / §10）。
  → 完了タスクが溜まってレーンが伸び続ける問題を防ぐ。能動ビュー（ボード）は常に未消化だけが見える。

### 3.3.1 クイック追加（未分類の一時保存）★v1.1

- ボード上部に**常時表示のクイック追加入力**。タイトルを打って Enter → 即タスク化。
- 作成時は**カテゴリを聞かない**（`category_id = null`）。未分類タスクは **「未分類」レーン**（ボード先頭）の `未着手` に入る。
- **未分類のまま完了してよい**。分類は任意で、強制しない（"あとで整理しなきゃ"の罪悪感を作らない）。
- 未分類レーンは軽い見た目で「整理を促すが急かさない」トーンにする。

### 3.3.2 タスクと予定の登録分離 ★v1.1

- **タスク追加**＝ボードのクイック追加から（→未分類/未着手）。
- **予定追加**＝カレンダー側の「＋予定」or 日付クリックから（→ `events`）。
- 両者は別エンティティ・別入口。UI上も混同させない。

### 3.3.3 タスク詳細パネル ★v1.1 / 更新v1.2

- カード/締切チップを**クリックで詳細を開く**。**右からのスライドオーバー（side-peek）**でボードを見せたまま編集。
- 編集項目：
  - タイトル
  - **メモ＝プレーンテキスト**（Markdownは不要・v1.2確定）
  - **リンク：URL＋ラベル（ラベルは任意）・複数可・クリックで開く**（v1.2確定）
  - **締切：プログレッシブ表示** ―「☐ 締切を設定」のチェックを入れた時だけ日付欄が出る。日付の隣に**任意の時刻**（15 分刻みプルダウン・「時刻なし」＝日付のみ）。★v1.5
  - カテゴリ、状態
- 作成/更新日時を表示。完了・削除アクション。
- **自動保存**（保存ボタンを置かない／§3.7）。操作ボタン（削除・完了戻し）は**入力項目の直下**に置く。★v1.5
- **不採用（ADR-0002 / 整理を生まない原則）**：優先度（高/中/低）・タスク/予定の手動色分けは「仕分け＝整理作業」を増やすため持たせない。区別は種別から自動で描く。
- 思想：**キャプチャは一行で速く、詳細は後から足す**。詳細入力を必須にしてキャプチャを遅くしない。

### 3.7 操作モデル（オーバーレイ駆動）★v1.2 ※最重要

「1画面・遷移ゼロ」を実現する操作の規律。**ベース画面（締切レーン＋ボード＋カレンダー）は常に固定**し、入力・編集は画面を**重ねて**行う。`prototype-overlay.html` が実挙動の正。

**オーバーレイの重さを3段階で使い分ける：**

| 重さ | 種類                               | 用途                                                |
| ---- | ---------------------------------- | --------------------------------------------------- |
| 最軽 | **その場編集（オーバーレイなし）** | タイトルのインライン編集など                        |
| 軽   | **ポップ（要素に紐づく小パネル）** | クイック追加、＋予定、日付選択、カテゴリ/状態の選択 |
| 中   | **スライドイン（右の side-peek）** | タスク詳細（メモ・リンク込み）                      |
| 封印 | ~~モーダル（中央・全面）~~         | ベースを覆うため原則使わない。破壊的確認のみ例外    |

**散漫でも壊れないためのルール：**

1. **自動保存**：保存ボタンを作らない。変更は即コミット（楽観的更新）。
2. **クリック外／Esc で即閉じる**：「保存しますか？」を出さない。
3. **オーバーレイは常に1枚だけ**：重ねない。新しく開く＝前のを閉じる。
4. **ドラッグ優先**：動かせるもの（状態・カテゴリ変更）はパネルを出さず直接操作。これが究極の「ボタン廃止」。

### 3.4 近日締切レーン

- 画面上部に**細い横ストリップ**として常時表示
- 締切（`due_date`）のある未完了タスクを**締切が近い順**に並べる
- **緊急度で色分け**（例：3日以内＝赤 / 7日以内＝橙 / それ以遠＝中立）
- 14日先の締切も今見える＝時間軸に埋もれない（これが「忘れない」の主担保）
- タップでタスク詳細へ

### 3.5 カレンダー（予定専用）★v1.4 改訂

> **v1.4 改訂（画面構成見直し / ADR-0001）**: 週/日ビュー切替と時間グリッド（ドラッグ移動・リサイズ）を廃止。狭い右カラムでも読みやすい**無限スクロールのアジェンダ（日次リスト）**に一本化した。

- 画面右に配置。接待・講演など**固定枠の予定**を扱う
- **アプリ内で予定の作成・編集・削除**ができる（v1）
- ビュー：**今日起点の無限スクロール・アジェンダ**。日付セクションを縦に連続表示し、未来方向はスクロールで自動継ぎ足し、過去方向は先頭の「前を表示」で読み込む。「今日へ」で今日へ復帰。
- 時間の変更は**予定詳細パネル（保存ボタン式／開始・終了の 15 分刻みプルダウン）**で行う（グリッド上のドラッグ編集は廃止）。日付選択はミニ月カレンダーで、**ドラッグ／2 クリックのどちらでも複数日**を選べる。★v1.5
- **締切は当日に表示する**（近日締切レーンの補助）。ただし**予定とは見た目で区別**する：予定＝時計アイコンの平たい行、締切タスク＝旗アイコン＋緊急度色の淡い塗りカード（色は種別・緊急度から自動。手動色分けはしない）。★v1.5
- 予定属性：タイトル、開始/終了日時、終日フラグ、場所（任意）、メモ（任意）

### 3.8 勤務地（その日の拠点）★v1.4 新設 / v1.5 表記統一

> **v1.4 新設（ADR-0001）**: 「その日はどこにいるか」を**予定とは別概念の日次ラベル**として持つ。終日予定では代用しない（用事と勤務地を混同させない）。
> **v1.5 表記統一**: 画面表記は **「勤務地」に一本化**（"シフト"の語は UI から除去。内部テーブル名 `shift_types`/`shifts` は据え置き）。「休み」も勤務地の一項目として登録する。

- マスタ（`shift_types`）＝勤務地のユーザー定義セット（例：渋谷店 / 恵比寿店 / りんご / 休み）。名前・色・並び順を「勤務地管理」side-peek で編集（カテゴリ管理と同型）。登録はマスタ制（その場の自由入力は持たない）。
- 割当（`shifts`）＝**1日1件**（`unique(owner_id, date)`）。複数勤務地の日は運用上1件にまとめる。
- 表示：アジェンダ各日の**日付の真下**に**色付き勤務地チップ**を縦置き（クリックでマスタから選択／解除）。左の色ドットは置かず、チップ自体の色で示す。★v1.5
- 繰り返し（毎週○曜＝△店 等）は v1 では実装しない（手動割当のみ）。

### 3.6 永続化

- すべての操作をバックエンド保存／楽観的UI更新／v1はリアルタイム同期なし

## 5. データモデル

### 5.1 方針

- **タスクと予定は別エンティティ**（`tasks` と `events`）。
- 近日締切レーン＝ `tasks.due_date` から描く。カレンダー＝ `events` から描く。
- 個人前提だが `owner_id` ＋ RLS を最初から入れ、将来のチーム化に備える。
- 予定テーブルに **Google同期用の箱を最初から（nullableで）持たせる**。v1では未使用。

### 5.2 テーブル定義（Postgres / Supabase 想定）

```sql
-- カテゴリ（スイムレーン）
create table categories (
  id          uuid primary key default gen_random_uuid(),
  owner_id    uuid not null references auth.users(id),
  name        text not null,
  position    text not null,            -- fractional index
  color       text,                     -- 任意
  created_at  timestamptz not null default now()
);

-- タスク
create table tasks (
  id           uuid primary key default gen_random_uuid(),
  owner_id     uuid not null references auth.users(id),
  category_id  uuid references categories(id) on delete set null,  -- null = 未分類
  title        text not null,
  description  text,                           -- メモ（自由記述）
  links        jsonb not null default '[]',    -- [{ "title": "...", "url": "..." }] ★v1.1
  status       text not null default 'todo',   -- 'todo' | 'doing'(=対応中フラグ) | 'done' ★v1.4
  position     text not null,                  -- カテゴリ内の並び順（v1.4: 状態列廃止により category 単位）
  due_date     date,                           -- 任意・日付。近日締切レーンの元 ★v1.1
  due_time     time,                            -- 任意・締切の時刻（null＝日付のみ。due_date がある時のみ意味を持つ）★v1.5
  completed_at timestamptz,                    -- 完了時刻
  archived_at  timestamptz,                    -- 完了＝即アーカイブ。30日経過で物理削除 ★v1.4
  created_at   timestamptz not null default now(),
  updated_at   timestamptz not null default now()
);

-- 予定（カレンダー）
create table events (
  id              uuid primary key default gen_random_uuid(),
  owner_id        uuid not null references auth.users(id),
  title           text not null,
  start_at        timestamptz not null,
  end_at          timestamptz not null,
  all_day         boolean not null default false,
  location        text,
  notes           text,
  -- ▼ v1.3: 繰り返し予定用の箱（v1では全てnull。実装はv1.5で行う）
  rrule           text,                           -- RFC 5545 繰り返しルール
  recurring_event_id uuid references events(id) on delete cascade, -- 例外回の親参照
  is_exception    boolean not null default false, -- この回だけの個別変更
  -- ▲
  -- ▼ 将来のGoogle同期用（v1では未使用・nullableで仕込むだけ）
  sync_source     text not null default 'local',  -- 'local' | 'google'
  google_event_id text,
  google_etag     text,
  last_synced_at  timestamptz,
  -- ▲
  created_at      timestamptz not null default now(),
  updated_at      timestamptz not null default now()
);

-- 勤務地・シフト種別のマスタ ★v1.4
create table shift_types (
  id          uuid primary key default gen_random_uuid(),
  owner_id    uuid not null references auth.users(id) on delete cascade,
  name        text not null,            -- 渋谷店 / 恵比寿店 / りんご / 休み
  color       text,                     -- チップ色（任意）
  position    text not null,            -- fractional index
  created_at  timestamptz not null default now()
);

-- 日へのシフト割当（1日1件） ★v1.4
create table shifts (
  id            uuid primary key default gen_random_uuid(),
  owner_id      uuid not null references auth.users(id) on delete cascade,
  date          date not null,
  shift_type_id uuid not null references shift_types(id) on delete cascade,
  created_at    timestamptz not null default now(),
  unique (owner_id, date)               -- 1日1シフト
);

-- 近日締切レーンの定番クエリを速く
create index idx_tasks_due on tasks (owner_id, due_date);
-- カレンダーの時間範囲クエリを速く
create index idx_events_start on events (owner_id, start_at);
-- シフト ★v1.4
create index idx_shift_types_owner on shift_types (owner_id, position);
create index idx_shifts_owner_date on shifts (owner_id, date);
```

### 5.3 設計上のキモ

- **並び順は fractional index（LexoRank的な文字列キー）**：整数indexの全件再採番を回避。挿入は1行更新で済む。
- **`updated_at` 全テーブル**：将来の競合解決（チーム化・Google同期）の布石。
- **RLS を最初から有効化**：`owner_id = auth.uid()` を各テーブルに設定。新規ユーザー登録時も自動的に制限が適用され、同一DB・同一テーブルでありながら完全に個人専用のセキュアな隔離環境が構築される。将来的に「ワークスペース」や「共有」機能を拡張する場合も、RLSの条件式を変更するだけでデータモデルを壊さずに移行可能。
- **未分類＝`category_id is null`** ★v1.1：専用テーブル不要。ボードは null カテゴリを「未分類」レーンとして描く。クイック追加は `category_id=null, status='todo'` で1行 insert するだけ。
- **リンクは JSONB（`links`）** ★v1.1：リンク単位での検索は不要なので別テーブルにせず軽量に。将来リンクに機能を持たせたくなったら別テーブルへ切り出す。
- **近日締切レーン**：`due_date is not null and status <> 'done' and archived_at is null` を `due_date asc` で取得。
- **完了＝即アーカイブ** ★v1.4：完了ゾーンへのドロップで `status='done', completed_at=now, archived_at=now` を一度に書く。ボード・締切レーンは `archived_at is null` でフィルタするため即座に消える。完了履歴は `archived_at is not null` を `completed_at desc` で表示。
- **30日物理削除** ★v1.4：`archived_at < now() - interval '30 days'` を **DELETE**（DBからも消す）。v1はアプリ起動時のフィルタ（クライアント sweep）で簡易実装。
- **シフト** ★v1.4：`shift_types`（マスタ）＋ `shifts`（1日1件割当）。カレンダー各日に色付きチップで表示。`shift_types` 削除時は `on delete cascade` で当該 `shifts` も消える。

## 6. 技術構成

| レイヤー     | 選定                                   | 備考                                                                                                                |
| ------------ | -------------------------------------- | ------------------------------------------------------------------------------------------------------------------- |
| フロント     | **Vite + React + TypeScript**          | 軽量かつ高速なSPA。SSR不要のためViteが最適                                                                          |
| バックエンド | **Supabase**（Postgres + Auth + RLS）  | DB・認証・マルチユーザー隔離                                                                                        |
| ホスティング | **Cloudflare Pages**                   | `dist/` を静的ホスティング。安価かつ高速                                                                            |
| 状態管理     | **TanStack Query**                     | クライアント側のキャッシュ、自動フェッチ、楽観的更新の実現                                                          |
| ルーティング | **なし（認証ゲーティング）**           | 1 画面・遷移ゼロのため URL ルータは未使用。`AuthContext` の状態でログイン画面 ↔ メイン画面を出し分ける。            |
| ボードDnD    | **dnd-kit**                            | カテゴリ単位リストの並べ替え・カテゴリ移動・完了ゾーンへのドロップ（v1.4: 状態列廃止）                              |
| カレンダー   | **自作（無限スクロール・アジェンダ）** | ライブラリに依存せず Tailwind でデザイン統一。v1.4 で時間グリッド（ドラッグ／リサイズ）を廃し、日次リストへ一本化。 |

### 6.1 カレンダー自作の方針

v1は「アプリ内で作成・編集」まで行うため、カレンダーUIは最大の作業ポイントとなる。

- カレンダーライブラリは採用せず、**Vite + React上でTailwind CSSを用いて完全に自作**する。
- ドラッグによる時間帯移動や、カード下端ドラッグによる時間枠の伸縮（リサイズ）は、マウス/タッチイベントを用いて自前で制御し、グリッド（例：15分または30分単位）へのスナップ処理を組み込む。
- これにより、外部ライブラリのスタイリングや機能的な制約に縛られず、軽量で拡張性の高いカレンダーコンポーネントを実現する。

## 9. 実装順序（v1）

各ステップで「毎日使える状態」を維持する。

1. **基盤**：Vite + React + TypeScript プロジェクト初期化、Tailwind CSS + shadcn/ui セットアップ
2. **Supabase**：スキーマ作成（繰り返し用カラム含む）、RLSポリシー設定によるマルチユーザー隔離の実装
3. **認証**：Supabase Authによるメール＋パスワードまたはマジックリンク認証と、ログイン制御の実装
4. **レイアウトシェル**（§3.7）：1画面レイアウト固定＋オーバーレイ基盤（side-peek／ポップ）の構築
5. **カテゴリ**：カテゴリの作成、リネーム、並べ替え、削除
6. **タスク**：タスクのCRUD＋クイック追加（未分類レーン）
7. **タスクボード**：カテゴリ×状態表示、dnd-kitによるドラッグ移動（状態・カテゴリの変更）
8. **タスク詳細**：side-peekパネルでの全項目編集・自動保存（デバウンス処理）
9. **近日締切レーン**：締切順クエリ、緊急度に応じた色分け、クリックで詳細パネル表示
10. **カレンダー基本**：週/日ビューの描画と、予定のCRUD（ポップによる入力）
11. **カレンダーDnD**：ドラッグでの時間移動、下端ドラッグによるリサイズ、グリッドスナップ
12. **自動アーカイブ**：完了タスクの一定期間経過後の自動アーカイブ処理
13. **楽観的更新**：TanStack Queryを用いた、通信待ちを感じさせないUI更新（UX仕上げ）
14. **Cloudflareデプロイ**：Pagesへのビルド・デプロイ設定
15. → v1.5：繰り返し予定（rrule.js + 繰り返しUI + 例外処理）
16. → 将来：Googleカレンダー双方向同期、チーム化（workspace・RLS拡張・Realtime）
