## 0. 北極星（設計の最上位原則）

**考えさせない・忘れさせない・整理させない。**
注意が逸れやすい状況でも回るよう、記憶や意志力に頼らせない。システム側が「今やること」と「迫る締切」を勝手に前へ押し出す。
→ この原則に反する機能（重い整理・手入力の多い計画機能）は採用しない。

## 1. コンセプト

1画面に3つの面を並置し、画面遷移なしで完結する：

1. **タスクボード**（左・メイン）… カテゴリ × 状態の2次元ボード。作業を個別登録 → ドラッグで消化。
2. **近日締切レーン**（上・常時表示）… 締切のあるタスクを締切順に常に見せ、埋もれを防ぐ。
3. **カレンダー**（右）… 接待・講演などの固定予定。v1はアプリ内で作成・編集まで。

タスク（こなす作業）と予定（出席する固定枠）は**別物として扱う**。締切は時間軸（カレンダー）に埋めず、常時表示のレーンで管理する。

### 1.1 差別化の中核 ★v1.2

**「すべてを1画面に、画面遷移なしで」** がこのプロダクトの存在意義。
既存ツール（Google Tasks / Todoist 等）は "リスト⇄ボード⇄カレンダー" を**タブで切り替える**設計。本プロダクトは切り替えを廃し、ベース画面を固定したまま情報を**スライドイン／ポップで重ねる**（§3.7）。
→ 新規性ではなく**「遷移ゼロで全状況が一目」というフィット**で価値を出す。注意が逸れやすいほど、画面遷移＝忘れる隙になるため、ここを最重要要件とする。

## 3. 機能要件（v1）

### 3.1 認証
- メール＋パスワード or マジックリンク（Supabase Auth）
- 自分のデータのみ閲覧・編集可（RLS）

### 3.2 カテゴリ
- 作成 / リネーム / 並べ替え / 削除（ユーザー定義：事務・経営・採用…）
- ボード上ではスイムレーン（横の帯）として並ぶ
- **カテゴリは折りたためる**（視界を絞れる）

### 3.3 タスクボード（カテゴリ × 状態）
- 縦＝カテゴリ（スイムレーン）／横＝状態（`未着手` / `対応中` / `完了`）
- タスクの作成（クイック追加）/ 編集 / 削除
- 属性：タイトル（必須）、メモ（任意）、**リンク（任意・複数可）**、カテゴリ（任意）、状態、**締切（任意・日付のみ）**、並び順
- DnD：
  - **横移動** = 状態変更
  - **レーンを跨ぐ移動** = カテゴリ変更
- **完了の自動アーカイブ**：`完了`にして一定期間（例：N日）経過したタスクはボードから自動的に畳む（アーカイブ）。別途アーカイブ一覧から参照可。
  → 完了タスクが溜まってレーンが伸び続ける問題を防ぐ。

### 3.3.1 クイック追加（未分類の一時保存）★v1.1
- ボード上部に**常時表示のクイック追加入力**。タイトルを打って Enter → 即タスク化。
- 作成時は**カテゴリを聞かない**（`category_id = null`）。未分類タスクは **「未分類」レーン**（ボード先頭）の `未着手` に入る。
- **未分類のまま完了してよい**。分類は任意で、強制しない（"あとで整理しなきゃ"の罪悪感を作らない）。
- 未分類レーンは軽い見た目で「整理を促すが急かさない」トーンにする。

### 3.3.2 タスクと予定の登録分離 ★v1.1
- **タスク追加**＝ボードのクイック追加から（→未分類/未着手）。
- **予定追加**＝カレンダー側の「＋予定」or 日付クリックから（→ `events`）。
- 両者は別エンティティ・別入口。UI上も混同させない。

### 3.3.3 タスク詳細パネル ★v1.1 / 更新v1.2
- カード/締切チップを**クリックで詳細を開く**。**右からのスライドオーバー（side-peek）**でボードを見せたまま編集。
- 編集項目：
  - タイトル
  - **メモ＝プレーンテキスト**（Markdownは不要・v1.2確定）
  - **リンク：URL＋ラベル（ラベルは任意）・複数可・クリックで開く**（v1.2確定）
  - **締切：プログレッシブ表示** ―「☐ 締切を設定」のチェックを入れた時だけ日付欄が出る（日付のみ／§10の任意時刻は別）
  - カテゴリ、状態
- 作成/更新日時を表示。完了・削除アクション。
- **自動保存**（保存ボタンを置かない／§3.7）。
- 思想：**キャプチャは一行で速く、詳細は後から足す**。詳細入力を必須にしてキャプチャを遅くしない。

### 3.7 操作モデル（オーバーレイ駆動）★v1.2 ※最重要

「1画面・遷移ゼロ」を実現する操作の規律。**ベース画面（締切レーン＋ボード＋カレンダー）は常に固定**し、入力・編集は画面を**重ねて**行う。`prototype-overlay.html` が実挙動の正。

**オーバーレイの重さを3段階で使い分ける：**

| 重さ | 種類 | 用途 |
|---|---|---|
| 最軽 | **その場編集（オーバーレイなし）** | タイトルのインライン編集など |
| 軽 | **ポップ（要素に紐づく小パネル）** | クイック追加、＋予定、日付選択、カテゴリ/状態の選択 |
| 中 | **スライドイン（右の side-peek）** | タスク詳細（メモ・リンク込み） |
| 封印 | ~~モーダル（中央・全面）~~ | ベースを覆うため原則使わない。破壊的確認のみ例外 |

**散漫でも壊れないためのルール：**
1. **自動保存**：保存ボタンを作らない。変更は即コミット（楽観的更新）。
2. **クリック外／Esc で即閉じる**：「保存しますか？」を出さない。
3. **オーバーレイは常に1枚だけ**：重ねない。新しく開く＝前のを閉じる。
4. **ドラッグ優先**：動かせるもの（状態・カテゴリ変更）はパネルを出さず直接操作。これが究極の「ボタン廃止」。

### 3.4 近日締切レーン
- 画面上部に**細い横ストリップ**として常時表示
- 締切（`due_date`）のある未完了タスクを**締切が近い順**に並べる
- **緊急度で色分け**（例：3日以内＝赤 / 7日以内＝橙 / それ以遠＝中立）
- 14日先の締切も今見える＝時間軸に埋もれない（これが「忘れない」の主担保）
- タップでタスク詳細へ

### 3.5 カレンダー（予定専用）
- 画面右に配置。接待・講演など**固定枠の予定**を扱う
- **アプリ内で予定の作成・編集・削除**ができる（v1）
- ビュー：**デフォルトは複数日（週相当）表示、特定日をクリックで日表示**に展開
- スクロールで前後の日へ移動
- **締切は表示しない**（時間軸に埋もれさせない方針）
- 予定属性：タイトル、開始/終了日時、終日フラグ、場所（任意）、メモ（任意）

### 3.6 永続化
- すべての操作をバックエンド保存／楽観的UI更新／v1はリアルタイム同期なし

## 5. データモデル

### 5.1 方針

- **タスクと予定は別エンティティ**（`tasks` と `events`）。
- 近日締切レーン＝ `tasks.due_date` から描く。カレンダー＝ `events` から描く。
- 個人前提だが `owner_id` ＋ RLS を最初から入れ、将来のチーム化に備える。
- 予定テーブルに **Google同期用の箱を最初から（nullableで）持たせる**。v1では未使用。

### 5.2 テーブル定義（Postgres / Supabase 想定）

```sql
-- カテゴリ（スイムレーン）
create table categories (
  id          uuid primary key default gen_random_uuid(),
  owner_id    uuid not null references auth.users(id),
  name        text not null,
  position    text not null,            -- fractional index
  color       text,                     -- 任意
  created_at  timestamptz not null default now()
);

-- タスク
create table tasks (
  id           uuid primary key default gen_random_uuid(),
  owner_id     uuid not null references auth.users(id),
  category_id  uuid references categories(id) on delete set null,  -- null = 未分類
  title        text not null,
  description  text,                           -- メモ（自由記述）
  links        jsonb not null default '[]',    -- [{ "title": "...", "url": "..." }] ★v1.1
  status       text not null default 'todo',   -- 'todo' | 'doing' | 'done'
  position     text not null,                  -- (category × status)セル内の並び順
  due_date     date,                           -- 任意・日付のみ。近日締切レーンの元 ★v1.1
  completed_at timestamptz,                    -- 完了時刻
  archived_at  timestamptz,                    -- 自動アーカイブ
  created_at   timestamptz not null default now(),
  updated_at   timestamptz not null default now()
);

-- 予定（カレンダー）
create table events (
  id              uuid primary key default gen_random_uuid(),
  owner_id        uuid not null references auth.users(id),
  title           text not null,
  start_at        timestamptz not null,
  end_at          timestamptz not null,
  all_day         boolean not null default false,
  location        text,
  notes           text,
  -- ▼ 将来のGoogle同期用（v1では未使用・nullableで仕込むだけ）
  sync_source     text not null default 'local',  -- 'local' | 'google'
  google_event_id text,
  google_etag     text,
  last_synced_at  timestamptz,
  -- ▲
  created_at      timestamptz not null default now(),
  updated_at      timestamptz not null default now()
);

-- 近日締切レーンの定番クエリを速く
create index idx_tasks_due on tasks (owner_id, due_date);
-- カレンダーの時間範囲クエリを速く
create index idx_events_start on events (owner_id, start_at);
```

### 5.3 設計上のキモ

- **並び順は fractional index（LexoRank的な文字列キー）**：整数indexの全件再採番を回避。挿入は1行更新で済む。
- **`updated_at` 全テーブル**：将来の競合解決（チーム化・Google同期）の布石。
- **RLS を最初から有効化**：`owner_id = auth.uid()`。将来はワークスペース判定に差し替え拡張。
- **未分類＝`category_id is null`** ★v1.1：専用テーブル不要。ボードは null カテゴリを「未分類」レーンとして描く。クイック追加は `category_id=null, status='todo'` で1行 insert するだけ。
- **リンクは JSONB（`links`）** ★v1.1：リンク単位での検索は不要なので別テーブルにせず軽量に。将来リンクに機能を持たせたくなったら別テーブルへ切り出す。
- **近日締切レーン**：`due_date is not null and status <> 'done' and archived_at is null` を `due_date asc` で取得。
- **自動アーカイブ**：`status = 'done'` かつ `completed_at < now() - interval 'N days'` を `archived_at` でマーク。v1はSupabaseのスケジュール関数 or アプリ起動時のフィルタで簡易実装。

## 6. 技術構成

| レイヤー | 選定 | 備考 |
|---|---|---|
| フロント | **Next.js (App Router) + TypeScript** | 既存スタック |
| バックエンド | **Supabase**（Postgres + Auth + RLS） | DB・認証・将来のリアルタイム/同期 |
| ホスティング | **Vercel** | 既存／無料枠で開始可 |
| 状態管理 | TanStack Query + 軽量クライアント状態 | 楽観的更新 |
| ボードDnD | **dnd-kit** | カテゴリ×状態の2次元ドラッグ（レーン＝droppable、状態セル＝ネストdroppable） |
| カレンダー | **ライブラリ採用を推奨**（後述） | 自前フル実装は沼 |

### 6.1 カレンダー実装の方針 ★要検討

v1は「アプリ内で作成・編集」までやるので、**カレンダーUIは v1 の最大の作業ポイント**。
**自前フル実装は避け、ライブラリを土台にすることを推奨**：

- 候補：`react-big-calendar`（MIT）、`@schedule-x/react`、FullCalendar（コアはMIT、一部ビューは有償）
- 評価軸：週/日ビューの有無、ドラッグでの作成/移動、スタイル自由度、ライセンス、**将来のGoogle同期との相性**
- 実装前に小さなスパイク（PoC）で1つ選定する

## 8. リスクと対策

| リスク | 深刻度 | 対策 |
|---|---|---|
| カレンダー（作成・編集付き）の実装が重い | **高** | 自前実装せずライブラリ採用。PoCで早期選定 |
| 「忘れない」を視覚のみに依存（v1は通知なし） | 中 | v1は常時表示の締切レーンで担保。アプリを開かない懸念があるなら **通知をv1.5で追加** |
| 2次元DnD（状態×カテゴリ）の挙動 | 中 | dnd-kitのネストdroppableで実装、早期に検証 |
| 完了タスクの溜まり | 中 | 自動アーカイブ＋カテゴリ折りたたみ |
| 並べ替えの再採番・将来の同期競合 | 中 | fractional index＋`updated_at` |
| 将来のGoogle双方向同期の競合処理 | （後回し） | スキーマに同期用カラムを仕込み済み。ロジックはv1.5+ |
| チーム化時のデータモデル破壊 | 中 | `owner_id`＋RLSを最初から、将来workspaceに拡張 |

## 9. 実装順序（v1）

各ステップで「毎日使える状態」を維持する。

1. **基盤**：Next.js + Supabase初期化、スキーマ作成、RLS設定
2. **認証**：サインイン／ログイン状態
3. **オーバーレイUIシェル**（§3.7）：固定ベース＋side-peek／ポップの土台。自動保存・クリック外閉じを最初に作り、以降のCRUDはこの上に乗せる
4. **カテゴリ CRUD ＋ 並べ替え**
5. **タスク CRUD**（タイトル・メモ・リンク・状態・締切）＋ クイック追加（未分類）
6. **タスクボード**：カテゴリ×状態の表示、dnd-kitで状態/カテゴリ変更
7. **タスク詳細 side-peek**：全項目編集・自動保存
8. **近日締切レーン**：締切クエリ＋緊急度の色分け＋クリックで詳細
9. **完了の自動アーカイブ**
10. **カレンダー PoC**：ライブラリ選定 → 予定の作成（ポップ）・編集・表示（週→日）
11. **楽観的更新＋仕上げ**
12.（v1完了。ドッグフーディング開始）
13. → v1.5：通知/リマインド、所要見積もり（任意）
14. → 将来：Googleカレンダー双方向同期、チーム化（workspace・RLS拡張・Realtime）

## 10. 未決事項（軽微・後で）

1. プロダクト名（仮称「Hibi」）
2. 自動アーカイブの日数 N（例：7日？）
3. 締切の緊急度しきい値（赤/橙の境目）
4. 認証方式：パスワード or マジックリンク
5. カレンダーライブラリの最終選定（PoC後）
6. リンクの上限数（v1は任意・上限ゆるめでよさそう／ラベルは任意で確定）
